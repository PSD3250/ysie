# 프로젝트 절대 원칙 (PROJECT_RULES.md)

이 파일은 프로젝트 진행 시 **AI가 반드시 준수해야 하는 절대 원칙**을 정의합니다.
이 원칙은 어떠한 기술적 이유보다 우선하며, 이를 어길 시 즉시 작업을 중단하고 시정해야 합니다.

> ⚠️ **경고: 이 파일을 읽지 않고 그 어떤 코드 수정도 진행하지 마십시오.**

---

## 1. 언어 원칙 (Language)
- 모든 아티팩트(`task.md`, `implementation_plan.md`, `walkthrough.md`) 및 작업 상태 메시지는 **반드시 한국어**로 작성합니다.
- 영어를 사용해야 할 경우(코드 변수명 등)를 제외하고는 사용자 소통은 100% 한국어로 합니다.

## 2. 작업 절차 원칙 (선 보고, 후 조치)
- 코딩이나 수정을 지시받으면 **반드시 수행 전에** 내용을 정리하여 사용자에게 확인(Confirm)을 받습니다.
- 사용자의 "진행해"라는 승인이 떨어지기 전에는 파일에 손대지 않습니다.

## 3. 코드 구조 원칙 (파일 분리)
- **HTML**(구조), **CSS**(디자인), **JavaScript**(기능/로직)를 철저히 분리합니다.
- 인라인 스타일 지양, 세 영역을 명확히 구분하여 작업합니다.

## 4. 외부 연동 원칙 (복붙 강조)
- 구글 앱 스크립트(GAS)나 기타 외부 프로그램에 코드를 복사해서 붙여넣어야 하는 경우, **반드시 별도로 강조**하여 사용자에게 명확히 알립니다.

## 5. 정직성 원칙 (모르면 모른다)
- 확실하지 않은 정보나 기억이 나지 않는 내용은 **절대로 아는 척하지 않습니다.**
- "확인해보겠습니다" 또는 "기억나지 않으니 다시 알려주세요"라고 솔직하게 말합니다.

## 6. 무결성 원칙 (Absolute Zero Tolerance)
- 오직 **100% 확신**만 허용됩니다. 0.0001%라도 불확실하면 멈추고 질문합니다.
- "아마 될 것이다(99.9%)"는 금지, 예외 없음.

## 작업 전 체크리스트

모든 작업 시작 전 아래를 마음속으로 복창합니다:

- [ ] 한국어로 작성했는가?
- [ ] 사용자에게 컨펌(Confirm)을 받았는가?
- [ ] HTML / CSS / JS 파일을 분리했는가?
- [ ] 외부 복붙 내용은 별도 강조했는가?
- [ ] 모르는 것을 아는 척하지 않았는가?
- [ ] `[MD] CANVAS ID STANDARD.md`의 최신 Canvas ID를 확인했는가?
- [ ] 파일 수정 시 PowerShell이 아닌 직접 편집 도구를 우선 사용했는가?
- [ ] `[MD] HISTORY.md`를 읽고 이전 작업 맥락을 파악했는가?

## 7. PowerShell 사용 원칙 (도구 안전성)
- 파일 수정은 **직접 편집 도구** (`replace_file_content`, `multi_replace_file_content`) **우선 사용**합니다.
- PowerShell은 **검색·확인 용도로만 최소화**합니다. (파일 읽기 등)
- **불가피하게 PowerShell로 파일을 수정해야 할 경우**, 사전에 사용자에게 명확히 고지하고 승인을 받습니다.
- 이유: PowerShell의 인코딩(`UTF8`, BOM 처리, 백틱 특수문자 등) 문제로 파일이 손상될 위험이 있음.

## 8. 히스토리 관리 원칙 (맥락 유지)
- 새 대화(창)가 시작되면, **어떤 작업이든 하기 전에** 반드시 아래를 먼저 수행합니다:
  1. `[MD] HISTORY.md`를 읽어 **최근 작업 흐름을 파악**
  2. 제공된 **이전 대화 요약(Conversation Summaries)** 확인
  3. 필요 시 **Knowledge Item(지식 저장소)** 참조
- **작업 완료 후** 반드시 `[MD] HISTORY.md`에 아래 형식으로 기록합니다:
  ```
  ## YYYY-MM-DD
  ### 작업 제목
  - **지시**: 사용자가 요청한 내용 요약
  - **수행**: AI가 수행한 내용 요약 (수정한 파일, 주요 변경사항)
  ```
- "기억이 없다"는 핑계는 금지 — **히스토리를 읽으면 됩니다.**

## 9. Git 커밋 원칙 (변경 이력 관리)
- 파일을 수정/생성/삭제한 작업이 완료되면, **반드시 Git 커밋**을 수행합니다.
- 커밋 메시지는 **한국어**로, 변경 내용을 간결하게 요약합니다.
  - 예: `"성적 수동 입력 학급 필드 수정"`, `"차트 폰트 크기 16으로 통일"`
- Git 실행 경로: `& 'C:\Program Files\Git\bin\git.exe'`
- 커밋 절차:
  1. `git add` — 변경된 파일 스테이징
  2. `git commit -m "변경 내용 요약"` — 커밋
- 이를 통해 문제 발생 시 **이전 상태로 되돌릴 수 있는 안전장치**를 확보합니다.
